## 追加应用对接

### 数据源

1. 来自最初的 `src/main/resources/apps/` 下的应用加载，此处应用程序还没赋予 UUID，所以不考虑对接基础
2. 数据库 `XApp` 的数据库记录（已存在应用数据）
3. 缓存目录 `???/apps/` 下的应用缓存，从数据库提取的应用级的核心数据信息

### 标识规则

1. 以当前入口应用环境变量 `Z_APP_ID` 为核心，通过 `WHERE APP_ID=? AND CODE=?` 进行唯一应用标识，参考现有环境变量的提取方法，BuildXxx 系列代码中已经存在。
2. 数据库数据优先，从数据库中不能加载时执行纯添加流程��若数据库中已存在数据，则和缓存中的 {UUID} 目录执行匹配
3. 匹配成功不再重新生成缓存，匹配不成功时才生成缓存

## 修复实现 (2026-03-02)

### 修改文件

1. **BuildApp.java**
   - 修改 `BuildMenuLoader.create()` 调用，传入 `vertx` 参数
   - 修改 `loadApps()` 和 `loadMenus()` 为异步调用，返回 `Future`
   - 调整 Future 链式调用结构

2. **BuildMenuLoader.java**
   - 添加 `Vertx` 实例字段和相关导入
   - 修改构造函数，接收 `vertx` 参数
   - 修改 `loadApps()` 方法为异步，返回 `Future<Void>`
   - 修改 `loadAppFromUri()` 方法为异步，返回 `Future<XApp>`
   - **核心修复**：实现数据库优先查询逻辑
     - 先从数据库查询（`WHERE APP_ID=? AND CODE=?`）
     - 使用 `Z_APP_ID` 环境变量（从 `globalConfig.appId` 获取）
     - 数据库存在则使用数据库 UUID
     - 数据库不存在才执行纯添加流程（yml id → instance.yml 映射 → 生成新 UUID）
   - 添加 `queryAppFromDatabase()` 方法，实现数据库查询

3. **BuildMenuPersister.java**
   - 修改 `generateMenuCache()` 方法，实现缓存匹配逻辑
   - 只在缓存不存在或内容不匹配时才重新生成
   - 匹配成功跳过写入，避免不必要的 I/O

### 核心逻辑

**应用加载流程：**
```
1. 读取 YAML 文件，提取 CODE
2. 获取 Z_APP_ID (globalConfig.appId)
3. 查询数据库: WHERE APP_ID=? AND CODE=?
4. 如果数据库存在：
   - 使用数据库的 UUID
   - 建立目录映射
5. 如果数据库不存在：
   - 检查 yml 中的 id
   - 检查 instance.yml 映射
   - 生成新 UUID
```

**缓存生成逻辑：**
```
1. 检查缓存文件是否存在
2. 如果存在，读取内容并比较
3. 内容匹配则跳过写入
4. 内容不匹配或不存在则重新生成
```
