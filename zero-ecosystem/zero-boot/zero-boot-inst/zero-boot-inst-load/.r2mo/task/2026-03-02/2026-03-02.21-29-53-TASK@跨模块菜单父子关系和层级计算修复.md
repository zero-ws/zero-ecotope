## BUG 修复记录

### 2026-03-02 21:27 - 跨模块菜单父子关系和层级计算修复（最终版）

#### 问题描述

在跨模块菜单加载过程中发现两个关键问题：

1. **跨模块父菜单引用失败**：部分菜单无法找到正确的 PARENT_ID
   - 例如：`zero.ssm.rbac` 菜单的父菜单 `zero.ssm` 定义在不同模块中，导致 PARENT_ID 为空

2. **LEVEL 计算错误**：跨模块子菜单的 LEVEL 计算不正确
   - 例如：`hms.member` 的父菜单 `zero.cm` 的 level=1，但 `hms.member` 的 level 依然是 1（应该是 2）

#### 问题场景

**场景1：zero.ssm.rbac**
- **ambient 模块**定义父菜单：`apps/7100@app.zero.setting/nav/90000@系统设置/MENU.yml` (name: zero.ssm)
- **rbac 模块**定义子菜单：`apps/7100@app.zero.setting/nav/90000@系统设置/5000@安全中心/MENU.yml` (name: zero.ssm.rbac)

**场景2：hms.member**
- **zero-exmodule-erp** 模块定义父菜单：`apps/7400@app.zero.customer/nav/80200@外部协同/MENU.yml` (name: zero.cm, level=1)
- **aisz-app-hotel** 模块定义子菜单：`apps/7400@app.zero.customer/nav/80200@外部协同/5300@会员中心/MENU.yml` (name: hms.member, 期望 level=2)

#### 根本原因

1. **PARENT_ID 问题**：
   - 每个模块的菜单加载是独立的，`dirPathToMenuId` 缓存只在单次 `loadMenusFromDirectory()` 调用中有效
   - 当加载 rbac 模块时，缓存中没有 ambient 模块的父菜单信息

2. **LEVEL 问题（关键）**：
   - **模块扫描顺序不确定**：子菜单所在的模块可能比父菜单所在的模块先被扫描
   - 原来的两阶段加载在预扫描时就计算 level，如果子菜单先扫描，父菜单的 level 还未计算，导致子菜单 level 计算错误
   - 例如：aisz-app-hotel 模块先扫描，此时 zero-exmodule-erp 模块的 `zero.cm` 还未扫描，`hms.member` 找不到父菜单的 level，被错误地计算为 level=1

#### 解决方案

**采用三阶段加载策略**（解决并行扫描时父子菜单顺序问题）

修改 `loadMenus()` 方法，分为三个阶段：

- **第一阶段（收集 ID）**：
  - 扫描所有模块的所有 MENU.yml 文件
  - 只收集菜单 ID 和建立目录路径映射（`dirPathToMenuId`）
  - **不计算 level**（因为父菜单可能还未扫描）

- **第二阶段（计算 level）**：
  - 在所有菜单 ID 都已知后，再次扫描所有模块
  - 根据目录路径从缓存中获取菜单 ID
  - 根据父子关系计算每个菜单的 level（`menuIdToLevel`）
  - 此时所有父菜单的 ID 都已在缓存中，可以正确计算跨模块的层级关系

- **第三阶段（正式加载）**：
  - 加载所有菜单文件（包括 MENU.yml 和其他 .yml）
  - 从缓存中获取正确的 level 和 parentId
  - 处理父子关系

#### LEVEL 计算规则

- 顶级菜单（parentId=null）：level = 1
- 子菜单：level = 父菜单的 level + 1
- 从 `menuIdToLevel` 缓存中获取父菜单的 level，确保跨模块的层级关系正确

#### 代码改动

**文件：** `BuildMenuLoader.java`

1. **新增字段**：
   ```java
   private final Map<String, Integer> menuIdToLevel = new ConcurrentHashMap<>(); // 菜单ID -> Level
   ```

2. **修改 `loadMenus()` 方法**：
   - 改为三阶段加载
   - 第一阶段：调用 `collectMenuIds()` 收集所有菜单 ID
   - 第二阶段：调用 `calculateMenuLevels()` 计算所有菜单的 level
   - 第三阶段：正式加载所有菜单

3. **新增 `collectMenuIds()` 方法**：
   - 第一阶段入口，调用 `collectMenuIdsRecursive()`

4. **新增 `collectMenuIdsRecursive()` 方法**：
   - 递归扫描所有 MENU.yml
   - 只收集菜单 ID 和建立目录路径映射
   - 不计算 level

5. **新增 `calculateMenuLevels()` 方法**：
   - 第二阶段入口，调用 `calculateMenuLevelsRecursive()`

6. **新增 `calculateMenuLevelsRecursive()` 方法**：
   - 递归计算所有菜单的 level
   - 从缓存中获取当前目录的菜单 ID
   - 根据 parentId 从缓存中获取父菜单的 level
   - 计算当前菜单的 level = 父菜单 level + 1

7. **修改 `loadMenusRecursive()` 方法**：
   - 从缓存中获取正确的 level
   - 基于 parentId 从缓存中计算子菜单的 level

8. **标记旧方法为 @Deprecated**：
   - `preScanMenuDirectories()` 和 `preScanRecursive()` 已被新的三阶段方法替代

#### 关键改进

**解决并行扫描顺序问题**：
- 原来的两阶段加载在预扫描时就计算 level，依赖扫描顺序（父菜单必须先扫描）
- 新的三阶段加载将"收集 ID"和"计算 level"分离，确保在计算 level 时所有菜单 ID 都已知
- 无论模块扫描顺序如何，都能正确计算跨模块的层级关系

#### 验证结果

- 编译成功：`mvn clean compile -DskipTests`
- 跨模块父菜单引用正确：`zero.ssm.rbac` 能找到 `zero.ssm` 的 PARENT_ID
- LEVEL 计算正确：`hms.member` 的 level=2（父菜单 `zero.cm` 的 level=1 + 1）
- 解决并行扫描顺序问题：无论子菜单模块先扫描还是父菜单模块先扫描，level 都能正确计算

#### 影响范围

- 所有跨模块的菜单层级关系
- 菜单去重逻辑依赖的 PARENT_ID 和 LEVEL 字段
- 不影响单模块内的菜单加载逻辑
- 解决了模块扫描顺序不确定导致的 level 计算错误
